# DailyTalk multi-speaker VITS recipe

This recipe trains and evaluates an English multi-speaker VITS model on
DailyTalk dialogue audio with ESPnet2.

Pretrained model: [espnet/dailytalk_VITS](https://huggingface.co/espnet/dailytalk_VITS)

The default setup uses raw 22.05 kHz audio, phoneme tokens generated by
`g2p_en_no_space`, and speaker IDs (`--use_sid true`).  DailyTalk contains
two speakers (`spk0` and `spk1`); the training configuration reserves speaker
ID 0 for `<unk>`, therefore `generator_params.spks` is set to 3.

## Recipe layout

| File or directory | Purpose |
| --- | --- |
| `run.sh` | Recipe entry point and default experiment options |
| `tts.sh` | ESPnet TTS pipeline implementation |
| `local/data.sh` | Downloads/prepares DailyTalk and creates data splits |
| `conf/tuning/train_vits.yaml` | Multi-speaker VITS training configuration |
| `conf/decode.yaml` | VITS decoding configuration |
| `conf/versa.yaml` | VERSA metric configuration used in Stage 9 |
| `cmd.sh` | Job-launcher configuration; defaults to Slurm |

## Prerequisites

Run commands from this directory after activating an ESPnet environment:

```bash
cd /path/to/espnet/egs2/dailytalk/tts1
conda activate espnet
```

Set `DAILYTALK` in `db.sh` to the directory that contains the DailyTalk
`data/` directory.  The default is:

```bash
DAILYTALK=downloads/dailytalk
```

The recipe's `cmd.sh` uses `slurm.pl`.  Adjust its CPU, memory, GPU, partition,
and account options for the target cluster before submitting jobs.

## Data preparation

`local/data.sh` creates deterministic dialogue-level splits after sorting
DailyTalk dialogue IDs numerically:

| Split | Default dialogue count |
| --- | ---: |
| `dev` | 200 |
| `eval1` | 200 |
| `tr_no_dev` | all remaining dialogues |

Prepare the corpus and Kaldi-style data directories:

```bash
bash local/data.sh --stage -1 --stop_stage 0
```

Stage `-1` downloads DailyTalk when needed.  To use an already prepared local
corpus, run only the split/preparation stage:

```bash
bash local/data.sh --stage 0 --stop_stage 0
```

## Full pipeline

Run all stages with the default configuration:

```bash
./run.sh --tag dailytalk_vits
```

The important TTS stages are:

| Stage | Operation |
| ---: | --- |
| 1 | Prepare ESPnet data directories |
| 2 | Format `wav.scp` |
| 3 | Prepare speaker IDs (`utt2sid`) |
| 4 | Filter unusable or overly long samples |
| 5 | Build the phoneme token list |
| 6 | Collect training statistics |
| 7 | Train VITS |
| 8 | Decode generated speech for `dev` and `eval1` |
| 9 | Score generated speech with VERSA |
| 10 | Package the model |

## Training

For a fresh experiment, run statistics collection and training:

```bash
./run.sh \
  --stage 6 \
  --stop_stage 7 \
  --tag dailytalk_vits
```

If statistics already exist, train only:

```bash
./run.sh \
  --stage 7 \
  --stop_stage 7 \
  --tag dailytalk_vits
```

The default model checkpoint for decoding is
`train.total_count.ave.pth` under `exp/tts_dailytalk_vits/`.

## Decoding

Generate speech for `dev` and `eval1`:

```bash
./run.sh \
  --stage 8 \
  --stop_stage 8 \
  --tag dailytalk_vits \
  --gpu_inference true \
  --inference_nj 2
```

`--gpu_inference true` makes Stage 8 and Stage 9 submit GPU jobs.  It should
remain enabled for GPU VERSA metrics even when an individual metric also has
`use_gpu: true` in `conf/versa.yaml`.

## VERSA evaluation

Run Stage 9 after Stage 8:

```bash
./run.sh \
  --stage 9 \
  --stop_stage 9 \
  --tag dailytalk_vits \
  --gpu_inference true \
  --inference_nj 2
```

Results for each evaluation set are written below:

```text
exp/tts_dailytalk_vits/decode_train.total_count.ave/<set>/scoring/versa_eval/
```

The default configuration in `conf/versa.yaml` currently enables:

- `mcd_f0`
- `discrete_speech`
- `sheet_ssqa`
- `pseudo_mos` with `utmos`
- `whisper_wer`
- `audiobox_aesthetics`
- `asr_match`
- `speaker`
- `asvspoof_score`

VERSA prints every metric enabled in `conf/versa.yaml`.  If `pseudo_mos` lists
`utmos`, `utmos` will appear in the final report even if another metric is the
main metric of interest.  Use a separate YAML file and pass it explicitly when
evaluating a different metric set:

```bash
./run.sh \
  --stage 9 \
  --stop_stage 9 \
  --tag dailytalk_vits \
  --gpu_inference true \
  --inference_nj 2 \
  --versa_config conf/versa_spk.yaml
```

### AASIST setup for `asvspoof_score`

`asvspoof_score` imports AASIST from VERSA's expected location:

```text
<espnet-root>/tools/versa/tools/checkpoints/aasist/
```

Install it once:

```bash
cd <espnet-root>/tools/versa
bash tools/install_asvspoof.sh
```

The metric's default checkpoint paths are relative to the current working
directory.  This recipe avoids that ambiguity by setting absolute
`model_config` and `model_path` values under the `asvspoof_score` entry in
`conf/versa.yaml`.  Update those values if this repository is moved.

### Re-running aggregation only

If VERSA jobs have finished but aggregation fails or needs to be repeated,
re-run aggregation without recomputing scores:

```bash
python pyscripts/utils/aggregate_eval.py \
  --logdir exp/tts_dailytalk_vits/decode_train.total_count.ave/eval1/scoring/versa_eval \
  --scoredir exp/tts_dailytalk_vits/decode_train.total_count.ave/eval1/scoring/versa_eval \
  --nj 2
```

Replace `eval1` with `dev` as needed.  The summarized metrics are written to
`avg_result.txt`; per-utterance JSON records are written to `utt_result.txt`.

`aggregate_eval.py` ignores a metric if that numeric field is absent from one
or more utterance records.  This prevents an incomplete optional metric (for
example, `asr_match_error_rate`) from terminating the entire aggregation.
